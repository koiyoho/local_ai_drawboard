# GitHub Online Update Design

## Goal

Add an admin-controlled online update flow for AI Board so a deployed instance can upgrade itself from GitHub without manual SSH file replacement.

The first production target is `aiboard.aipowers.site`. The preferred update source is GitHub Releases, using immutable release packages generated by the existing deploy package pipeline.

## Non-goals

- Do not auto-apply updates silently.
- Do not replace the existing Vite, React, Fastify, Prisma, or deployment architecture.
- Do not overwrite local data, uploaded assets, generated images, exports, `.env`, or SQLite files.
- Do not use branch pull/build as the default production upgrade path.
- Do not implement PSD export in this update feature. PSD export is recorded separately in `docs/ai-board-feature-roadmap.md`.

## Current State

The project already has check-only update foundations:

- `GET /api/system/version`
- `GET /api/system/update/check`
- `server/system-update.ts`
- `src/components/SystemUpdatePanel.tsx`
- `npm run deploy:package`
- `scripts/write-server-package.cjs`

The missing part is a safe apply/install flow.

## Recommended Approach

Use GitHub Releases mode first.

Each release publishes:

- a runtime package generated by `npm run deploy:package`,
- a manifest JSON,
- a sha256 checksum,
- release notes,
- channel metadata such as `stable` or `local`.
- migration metadata such as `migrationMode`.

The app checks the manifest, lets an admin review the target version, then starts an update job. Fastify only creates the job, snapshots the manifest, and launches a detached updater. The updater owns every mutating step: download, checksum verification, staging, backup, install, restart, health check, and rollback.

Branch mode can remain a later admin-only advanced option for private deployments. It is riskier because build failures happen on the installed server.

The first implementation supports public GitHub Releases only. Private GitHub Releases require token storage, request-header handling, log redaction, and permission copy, so they are deferred.

## Architecture

### Server API

Add these admin-only endpoints:

- `POST /api/system/update/apply`
- `GET /api/system/update/jobs/:id`

`POST /api/system/update/apply` should:

1. Re-run update check.
2. Reject if no update is available.
3. Reject if manifest channel does not match current channel.
4. Reject if the manifest URL or package URL is not from an allowed release host.
5. Reject if the request body includes `confirmedVersion` and it does not equal the latest manifest version.
6. Reject if the manifest declares an unsupported migration mode.
7. Reject if another update job is active.
8. Create an update job file under `tmp/updates`.
9. Write the manifest snapshot used by the updater.
10. Start the updater handoff.
11. Return `{ jobId }` immediately.

The first-version request body is optional. If provided, it may include:

```json
{
  "confirmedVersion": "1.1.0"
}
```

The server remains authoritative: it always re-checks the manifest and only uses `confirmedVersion` to prevent applying a different version than the one the admin reviewed.

`GET /api/system/update/jobs/:id` should return the current update job state.

### Job State

Use file-based state for the first version so the updater process and Fastify server can share progress without introducing a database migration.

Suggested path:

- `tmp/updates/current-update.json`
- `tmp/updates/jobs/<jobId>.json`
- `tmp/updates/update.lock`
- `tmp/updates/jobs/<jobId>-manifest.json`

Suggested state fields:

- `id`
- `status`
- `step`
- `fromVersion`
- `toVersion`
- `channel`
- `startedAt`
- `updatedAt`
- `message`
- `error`
- `rollbackStatus`
- `updaterPid`
- `lockExpiresAt`

Allowed statuses:

- `queued`
- `downloading`
- `verifying`
- `backing_up`
- `staging`
- `installing`
- `restarting`
- `health_checking`
- `completed`
- `failed`
- `rolled_back`

Concurrency and recovery rules:

- Creating an update job must acquire `tmp/updates/update.lock` atomically.
- Atomic lock creation means an exclusive file create/open operation such as Node `fs.open` with `wx`. Do not implement this as "check if file exists, then write".
- The lock stores `jobId`, `updaterPid`, `startedAt`, and `lockExpiresAt`.
- A new update is rejected while the lock points to an active job.
- If the server restarts and finds a stale lock whose `lockExpiresAt` has passed and whose job is not in a terminal state, it marks the job `failed` with a recovery message instead of starting a second updater.
- Terminal states are `completed`, `failed`, and `rolled_back`.
- Manual recovery should be possible by deleting the lock only after confirming no updater process is still running.

### Updater Process

The running Node process must not overwrite or restart itself directly.

For Linux/systemd deployment:

1. Fastify creates the job and writes the manifest snapshot.
2. Fastify spawns a detached updater script.
3. The updater downloads the package from the manifest snapshot into `tmp/updates/packages`.
4. The updater verifies sha256.
5. The updater creates timestamped runtime and data backups.
6. The updater extracts into a staging directory.
7. The updater validates package contents.
8. The updater swaps runtime files while preserving protected local paths.
9. The updater runs only fixed, project-owned install/init steps.
10. The updater restarts `tldraw-ai-board.service`.
11. The updater polls the health check endpoint.
12. The updater marks the job complete or restores the backup and marks rollback result.

For Windows/local desktop, use the same job model but a Windows-specific updater script later because locked files and service control differ.

### Backup and Rollback

Runtime and data rollback must be designed separately.

Runtime backup:

- Create `tmp/updates/backups/<jobId>/runtime`.
- Copy only the currently deployed application runtime before swapping package contents: server code, built client assets, package metadata, scripts, Prisma schema/migration files, and service-managed runtime files.
- Exclude all protected local paths, local data directories, generated media, uploads, exports, and transient update files from the runtime backup.
- Restore the runtime backup if install, restart, or health check fails.

Data backup:

- Create `tmp/updates/backups/<jobId>/data`.
- Snapshot SQLite files before any migration or database initialization step using a consistent backup method.
- If using direct file snapshots, stop the service first or otherwise ensure no writer is active, then include the main SQLite file plus `*.db-wal` and `*.db-shm` sidecar files when present.
- A future SQLite backup API implementation is acceptable, but the first implementation must not copy a live SQLite database while the service may still be writing to it.
- Preserve uploaded assets, generated images, exports, and configured local storage directories in place; do not move or replace them during upgrade.
- If a database migration fails, restore the SQLite snapshot before marking the job failed.
- If a migration is irreversible, the updater must reject the package before applying it in the first implementation.

Migration policy:

- The manifest must include `migrationMode`.
- Allowed first-version values are `none`, `reversible`, and `manual_required`.
- `none` means no database migration is required.
- `reversible` means the updater may proceed after creating a SQLite snapshot.
- `manual_required` means automatic apply is blocked and the UI must show manual upgrade guidance.
- Unknown values are rejected.

Rollback result must be recorded in the job state as one of:

- `not_needed`
- `restored_runtime`
- `restored_runtime_and_data`
- `failed_manual_recovery_required`

Rollback must also record a machine-readable failure reason. Required first-version reason categories:

- `backup_failed`
- `install_failed`
- `migration_failed`
- `restart_failed`
- `health_check_failed`
- `version_mismatch`
- `rollback_failed`

If the service cannot be restarted after rollback, the updater must leave the latest job state and recovery notes on disk under `tmp/updates/jobs/<jobId>.json` and `tmp/updates/jobs/<jobId>-recovery.md` so SSH/manual recovery has a known starting point.

The updater must write or refresh the recovery file before each destructive phase begins: before runtime swap, before database migration, before service restart, and before rollback. The file should include the job id, source version, target version, backup paths, active phase, last successful phase, failed command if any, and the manual restore command outline. This ensures recovery guidance exists even if the updater process exits abruptly.

### Protected Paths

The updater must not overwrite or delete:

- `.env`
- `prisma/*.db`
- `public/uploads`
- `generated-images`
- `exports`
- `tmp/updates`
- any configured local storage directory

Package validation should reject archives that contain path traversal entries or writes outside the intended runtime directory.

The archive must also be rejected when it contains protected paths directly. Protected local data should survive because it is excluded from the package swap, not because package files win over local files.

The archive must contain a package manifest such as `deploy-manifest.json`. The updater rejects the package if the manifest is missing, malformed, or inconsistent with the update manifest.

Required package manifest fields:

- `appName`
- `version`
- `commit`
- `channel`
- `migrationMode`
- `entrypoint`
- `files`

The package manifest must match the update manifest for `version`, `commit`, `channel`, and `migrationMode`. The implementation must define the manifest as a typed schema in server/updater source code, for example with zod, and cover it with parser tests and invalid-fixture tests. The release package generator must use the same schema or a shared schema-derived writer so the producer and consumer cannot drift silently. The first-version schema should be equivalent to:

```json
{
  "appName": "tldraw-ai-board",
  "version": "1.1.0",
  "commit": "abc123",
  "channel": "stable",
  "migrationMode": "none",
  "entrypoint": "dist/server/index.js",
  "files": [
    {
      "path": "dist/server/index.js",
      "size": 12345,
      "sha256": "hex-encoded-sha256"
    }
  ]
}
```

`files` entries must include relative paths, file sizes, and sha256 checksums for package contents. File paths must be relative, normalized, and must not include protected paths.

The release package must not define arbitrary install, postinstall, update, or migration shell commands. The first implementation only allows fixed project-owned steps, for example:

- install production dependencies with the repository's chosen package manager,
- generate Prisma client if required by the packaged runtime,
- run the project's existing database initialization or migration command when `migrationMode` allows it.

Any package-provided script name, shell command, or hook is ignored or rejected.

The updater must choose dependency and initialization commands from the installed application code, not from release metadata. These commands must be represented in code as an explicit allowlist, not assembled from manifest fields or package-provided script names. For the current project, the first implementation should derive the package manager from the committed lockfile and use a fixed command path such as:

- `npm ci --omit=dev` when `package-lock.json` is present,
- `npx prisma generate` only if Prisma client generation is required by the packaged runtime,
- the existing project-owned database initialization command only when `migrationMode` is `reversible`.

If the expected lockfile or project-owned command is missing, the updater must fail before swapping runtime files.

### Health Check

The first implementation should use a concrete health endpoint:

- `GET /api/system/health`

The endpoint should return HTTP 200 only when:

- the server process is alive,
- the database is readable,
- the runtime version can be loaded.

The updater should poll this endpoint after restart with a bounded timeout. After the health endpoint returns HTTP 200, the updater should call `GET /api/system/version` and verify that the reported version equals the update job target version. A health timeout or version mismatch is treated as update failure and triggers rollback when possible.

### Admin UI

Extend `SystemUpdatePanel` from check-only to full update control:

- Show current version, commit, build time, variant, and channel.
- Show latest release version, commit, publish time, and notes.
- Show whether package verification metadata exists.
- Show one explicit primary action: `下载并升级`.
- Show progress steps while the updater runs.
- Explain that the service will restart and the page may disconnect briefly.
- Show failure reason and rollback status if the update fails.

The board menu upgrade button should open this panel or the admin page. It should not execute updates directly from the board menu.

## Error Handling

The update flow should fail closed:

- missing manifest: show configuration error,
- wrong channel: reject,
- older or equal version: reject apply,
- missing sha256: reject apply,
- checksum mismatch: reject and keep current version,
- invalid archive: reject and keep current version,
- install failure: rollback when possible,
- restart or health check failure: rollback when possible,
- rollback failure: preserve logs and show manual recovery guidance.

The UI must show status with text and structure, not color alone.

## Security

- All apply and job detail endpoints require admin authentication.
- Manifest package URL must be HTTPS in production.
- The first implementation only accepts manifest URLs and package URLs from configured allowed hosts. The default allowed hosts should cover verified GitHub Release hosts for the actual download flow, such as `github.com`, `objects.githubusercontent.com`, and `release-assets.githubusercontent.com`.
- `UPDATE_PACKAGE_ALLOWED_HOSTS` can narrow the allowed host list for self-hosted deployments.
- `UPDATE_MANIFEST_ALLOWED_HOSTS` can narrow the manifest host list when the manifest is served separately.
- Package downloads must validate every redirect hop and the final resolved URL host against `UPDATE_PACKAGE_ALLOWED_HOSTS`. The updater should either disable automatic redirects and validate hop by hop, or use an HTTP client that exposes redirect targets for validation.
- sha256 verification is required.
- Signature verification is optional for the first implementation but should be supported when `UPDATE_PUBLIC_KEY` is configured.
- The first Linux/systemd implementation may run from the existing root-managed deployment environment. The production hardening target is a dedicated updater user or sudoers rule that can write only to `/srv/tldraw-ai-board`, its update work directories, and restart only `tldraw-ai-board.service`.
- Release metadata must not expose secrets.
- Private GitHub Release tokens are not supported in the first implementation. If added later, tokens must stay in server-only environment variables and must never be written to job state, logs, release notes, or browser payloads.

## Release Flow

Recommended release process:

1. Merge changes to the release branch.
2. Run tests and build.
3. Run `npm run deploy:package`.
4. Generate `deploy-manifest.json` with package metadata and file checksums.
5. Validate the package manifest against the archive contents.
6. Compute sha256 for the final release package.
7. Publish GitHub Release assets.
8. Publish/update the manifest URL.
9. Admin opens update panel and applies the update.

## First Implementation Slice

Build this in small steps:

1. Add update job state helpers.
2. Add package download and sha256 verification helpers.
3. Add `POST /api/system/update/apply`.
4. Add `GET /api/system/update/jobs/:id`.
5. Add `GET /api/system/health`.
6. Add Linux updater script for `/srv/tldraw-ai-board` and `tldraw-ai-board.service`.
7. Extend `SystemUpdatePanel` with apply and progress polling.
8. Wire board upgrade button to open the update panel or admin route.
9. Add tests for manifest rejection, wrong channel, no update, confirmed-version mismatch, active job, stale lock, atomic lock contention, manifest host rejection, package host rejection, redirect host rejection, checksum mismatch, package manifest mismatch, arbitrary script rejection, unsupported migration mode, protected path rejection, and job state transitions.

## Verification Plan

Run:

```powershell
npm run build
npm run test
```

Manual checks:

- Admin can see current and latest version.
- Wrong-channel manifest is rejected.
- Missing checksum blocks apply.
- Apply creates a job and returns quickly.
- UI polls progress without layout breakage.
- Service restart recovers the app.
- Failed health check rolls back or reports manual recovery clearly.
- Archive path traversal is rejected.
- Package attempts to overwrite protected paths are rejected.
- Stale update locks are reported instead of starting duplicate update jobs.
- Database snapshot is restored when migration fails.
- SQLite snapshots include WAL/SHM sidecar files when present.
- `manual_required` migration packages are blocked from automatic apply.
- Health check success is followed by version verification.
- Redirected package downloads are rejected when any hop leaves the allowed host list.
- Package manifest metadata and file checksums match the archive contents.
- Package-provided arbitrary install scripts are rejected.
- `confirmedVersion` mismatch blocks apply.

## First-version Defaults

- Manifest URL is configured by `UPDATE_MANIFEST_URL`; the first implementation assumes a public GitHub Release or public static manifest URL.
- Production service name is `tldraw-ai-board.service`.
- First implementation targets Linux/systemd only. Windows/local desktop updater support is deferred.
- sha256 verification is required. Signature verification is optional and only enabled when `UPDATE_PUBLIC_KEY` is configured.
- Branch pull/build update mode is deferred.
