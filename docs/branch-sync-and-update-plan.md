# Branch sync and online update plan

This project ships two variants from one codebase:

- `main`: complete multi-user build with authentication, user review, usage controls, asset integrity tooling, API settings, Codex login, and online update management.
- `local_board`: single-user local build. It keeps board features, API settings, Codex login, and online update management. It does not expose login, registration, user review, usage management, or asset integrity management.

## Branch policy

`main` is the feature trunk. All shared product work lands on `main` first.

`local_board` is a long-lived downstream branch. Its permanent differences must stay limited to:

- environment defaults for local single-user mode,
- documentation that is specific to local deployment,
- optional packaging or release-channel metadata.

Do not implement shared board features directly on `local_board`. If a feature is useful in both variants, implement it on `main`, verify it there, then merge `main` into `local_board`.

Recommended sync flow:

```powershell
git switch main
git pull --ff-only origin main
npm run test

git switch local_board
git merge main
npm run test
git push origin local_board
```

If conflicts repeat in the same files, reduce the branch-only changes instead of accepting long-term manual conflict resolution.

## Runtime variant model

The application reads `APP_VARIANT`.

- `APP_VARIANT=main` or an unset value means the full multi-user build.
- `APP_VARIANT=local` means the local single-user build.

In local mode:

- the server returns a deterministic local user from auth helpers,
- board, asset, provider setting, and prompt recipe ownership continues to use `userId`,
- the local user is created or updated during database initialization,
- login and registration routes are disabled,
- front-end login redirects are bypassed,
- admin UI only renders API settings, Codex login, and online update panels.

This keeps most business routes identical across both variants.

## Admin center modules

The admin shell is shared. Modules are selected by variant:

| Module | main | local_board |
| --- | --- | --- |
| API settings | Yes | Yes |
| Codex login | Yes | Yes |
| Online update | Yes | Yes |
| User review | Yes | No |
| Usage management | Yes | No |
| Asset integrity | Yes | No |

## Online update system

Both variants use the same update API with different release channels.

- `main` uses `UPDATE_CHANNEL=stable` by default.
- `local_board` uses `UPDATE_CHANNEL=local` by default.

The server reads a JSON manifest from `UPDATE_MANIFEST_URL`. The manifest format is:

```json
{
  "channel": "local",
  "version": "0.2.0",
  "commit": "abc123",
  "publishedAt": "2026-05-15T00:00:00Z",
  "packageUrl": "https://example.com/tldraw-ai-board-local-0.2.0.tar.gz",
  "sha256": "hex-encoded-sha256",
  "signature": "optional-signature",
  "minVersion": "0.1.0",
  "releaseNotes": "Short release notes"
}
```

Initial implementation requirements:

- expose current version, variant, and channel,
- check the manifest,
- reject manifests for the wrong channel,
- report whether a newer version is available,
- avoid applying updates unless manifest and package verification are available.

Full apply/update requirements before production use:

- download update package into `tmp/updates`,
- verify `sha256`,
- verify signature when signing is configured,
- backup current runtime files and SQLite data,
- stage and verify the extracted package,
- use a separate updater process on Windows because the running Node process cannot safely overwrite itself,
- restart the service,
- rollback if the new service fails its health check.

### GitHub-based auto update flow

The preferred distribution source is GitHub, not ad-hoc server file copying. The app should support two compatible GitHub update modes:

1. **GitHub Releases mode**
   - Each release publishes a runtime package generated by `npm run deploy:package`.
   - The release also publishes or references an update manifest JSON.
   - The manifest includes version, channel, commit, package URL, sha256, minimum supported version, and release notes.
   - This is the recommended mode for production because the installed app downloads immutable release assets.

2. **GitHub branch mode**
   - The updater checks a configured branch such as `main` or `local_board`.
   - It compares the local commit with the remote commit.
   - It can pull source, build locally, and restart the service.
   - This mode is useful for private/self-hosted deployments but is riskier because build failures happen on the user machine.

For `aiboard.aipowers.site`, use GitHub Releases mode for stable upgrades. Branch mode can remain an admin-only advanced option.

### Production release hosting for aiboard.aipowers.site

The production updater currently downloads the manifest and runtime package through the app domain:

```text
https://aiboard.aipowers.site/update-release/v<version>/update-manifest-v<version>.json
https://aiboard.aipowers.site/update-release/v<version>/aiboard-runtime-v<version>-<commit>.tgz
```

These files must live outside the deployed app runtime because an online update replaces `dist`, `package.json`, service files, and other runtime paths. Do not place update manifests or packages under `dist/client`; those files can disappear during the next package swap and may be served as the SPA fallback HTML instead of JSON.

Current production convention:

- persistent host directory: `/srv/tldraw-ai-board-updates/releases/`
- nginx alias: `/update-release/` -> `/srv/tldraw-ai-board-updates/releases/`
- app install directory: `/srv/tldraw-ai-board`
- service: `tldraw-ai-board.service`
- optional Gemini Bridge service: `tldraw-ai-board-gemini-bridge.service`
- service port: `3333`
- SSH alias: configure a local `aiboard-prod` host entry outside this repository; use SSH key auth and never commit host credentials or passwords.

When publishing a release for production update testing:

1. Build and package locally.
2. Upload or copy the versioned manifest and package to `/srv/tldraw-ai-board-updates/releases/v<version>/`.
3. Set `UPDATE_MANIFEST_URL` in the production `.env` to the versioned `/update-release/` manifest URL.
4. Ensure `UPDATE_MANIFEST_ALLOWED_HOSTS` and `UPDATE_PACKAGE_ALLOWED_HOSTS` include `aiboard.aipowers.site` when using the app-domain release host.
5. Restart `tldraw-ai-board.service` after changing `.env`.
6. Apply the update from the admin UI or update API.

The updater restarts the main app service first, waits for health/version checks, then runs `systemctl try-restart tldraw-ai-board-gemini-bridge.service` by default. This keeps Gemini Web Bridge code and environment settings in sync after a runtime package upgrade without failing deployments that do not install the bridge service. Set `UPDATE_RESTART_GEMINI_BRIDGE=0` to disable this step, or `UPDATE_GEMINI_BRIDGE_SERVICE_NAME=<service>.service` if a deployment uses a different bridge unit name.

The GitHub Release can still hold the immutable public release assets, but the self-hosted `/update-release/` path is the operational source for this deployment until GitHub asset fetching is proven reliable for the production server.

### Apply update endpoint

The existing API already supports:

- `GET /api/system/version`
- `GET /api/system/update/check`

The next implementation should add an admin-only apply endpoint:

- `POST /api/system/update/apply`

The apply endpoint should:

1. Check that an update is available.
2. Reject wrong release channels.
3. Download the package into `tmp/updates`.
4. Verify sha256 before extraction.
5. Optionally verify signature when `UPDATE_PUBLIC_KEY` is configured.
6. Create a timestamped backup of runtime files and SQLite data.
7. Extract into a staging directory.
8. Validate package contents against the deploy manifest rules.
9. Run production dependency install if needed.
10. Run database initialization/migration compatibility step.
11. Hand off restart to an external updater process.
12. Health check the restarted app.
13. Try to restart the optional Gemini Bridge service after the app health/version checks pass.
14. Roll back backup if health check fails.

The endpoint should return an update job id immediately instead of keeping the HTTP request open through the full install.

### Updater process

The running Node server should not overwrite or restart itself directly. Use a separate updater process:

- Linux/systemd: spawn a detached Node script or shell script that runs after the API response is returned, then calls `systemctl restart tldraw-ai-board.service`.
- Windows/local desktop: spawn a separate updater executable or Node script because the running process may lock files.

The updater should write progress to a small local state file such as `tmp/updates/current-update.json`:

```json
{
  "id": "update-20260516-001",
  "status": "downloading",
  "fromVersion": "1.0.0",
  "toVersion": "1.1.0",
  "startedAt": "2026-05-16T00:00:00Z",
  "message": "Downloading release package"
}
```

Additional status endpoint:

- `GET /api/system/update/jobs/:id`

### Admin UI behavior

The existing `SystemUpdatePanel` should evolve from check-only to:

- show current version, channel, build time, and commit,
- show latest GitHub release metadata,
- show release notes,
- show package verification status,
- provide `下载并升级` when an update is available,
- show step progress: checking, downloading, verifying, backing up, installing, restarting, health checking, complete,
- warn that the app will be temporarily unavailable during restart,
- show rollback result if the update fails.

The board global menu upgrade button can open the same system update panel or route to the admin settings page. It should not run updates silently from the board menu.

### Safety rules

- Update apply must require an admin user.
- The update package must never overwrite `.env`, `prisma/*.db`, `public/uploads`, `generated-images`, or local export directories.
- The app must keep a rollback package until the new version passes health check.
- The update job should refuse to start if another update is running.
- The UI must make it clear whether the user is on `stable`, `local`, or another channel.
- Auto-download can be enabled later, but auto-apply without explicit admin approval should remain disabled by default.

## Version metadata

Builds should expose a runtime version payload containing:

- package version,
- current git commit when available,
- variant,
- update channel,
- build time.

The server can fall back to `package.json` and environment variables if build metadata is not present.

## Verification expectations

For shared changes on `main`:

```powershell
npm run test
```

For local mode before pushing `local_board`:

```powershell
$env:APP_VARIANT="local"
npm run test
```

At minimum, local mode must verify:

- home route can create/open a board without login,
- `/api/auth/me` returns the local user,
- `/api/auth/login` and `/api/auth/register` are disabled,
- admin payload excludes user review, usage, and asset integrity requests,
- update endpoints return version and manifest status.
